## 整理知識卡（2-B 多步驟私訊草稿）

| 項目 | 內容 |
|------|------|
| 顧問想做的事 | 新增或整理單張知識卡草稿，送 Admin 審核 |
| 如何開始 | 私訊 `幫我整理知識卡`，依提示提供內容 |
| 如何補充 | `補充：…` |
| 如何修改 | `修改：…` |
| 如何重新整理 | `重新整理`（基於既有草稿，不腦補） |
| 如何轉成 JSON | `轉成 JSON`（選用，須通過 validator） |
| 如何確認送出 | `確認送出`（同一 transaction 寫入 `pending_knowledge_reviews` 並標記 session submitted） |
| 如何取消 | `取消` / `停止整理` / `先不用` |
| 如何完成 | `完成`（保留資料，清除 active session） |
| 小助手動作 | 產出人類可讀草稿（需 OPENAI_API_KEY）；不直接寫入 `knowledge_cards` |
| 是否公開回群組 | 否（私訊回覆） |
| 是否需要確認 | 顧問需「確認送出」；Admin 需「確認更新 K-xxx」才正式上線 |

**權限：** Consultant 只能產草稿 + 確認送出，**不能**確認更新、暫停或恢復知識卡。Consultant 輸入「確認更新」會被拒絕。

**過期：** 超過 24 小時（可由 `DM_SESSION_TIMEOUT_HOURS` 調整）未操作，下次私訊時自動過期，需重新輸入「幫我整理知識卡」。

**2-C-1 支援私訊截圖：** 也可直接貼截圖，小助手會先理解截圖內容再整理成草稿（需 OPENAI_API_KEY）。截圖不會被保存，僅保留文字摘要。

**不明輸入：** 有 active session 時，「好 / 了解 / 謝謝」等不會觸發整理；無 active session 時至多一次簡短提示。

## Admin 審核顧問草稿

| 指令 | 行為 |
|------|------|
| `確認更新 K-YYYYMMDD-XX` | validator 通過後寫入 `knowledge_cards`，待審 status → approved |
| `需要修改 K-xxx：…` | 記錄 admin_response，推送修改意見給顧問，不寫入 DB |
| `退回 K-xxx` | status → rejected，通知顧問 |

多筆待審時必須指定短碼；單筆可省略。可回覆草稿訊息 + `確認更新`（quotedMessageId 輔助）。

## 使用說明

私訊觸發：`使用說明`、`help`、`顧問手冊`、`小助手說明`、`你會做什麼`  
群組觸發（admin / consultant）：同上，回群組版簡要說明。  
`我的層級` 維持獨立身份查詢，不觸發使用說明。

## 代回群組

| 項目 | 內容 |
|------|------|
| 顧問想做的事 | 將顧問私訊內容逐字代回群組 |
| 建議標準語法 | 私訊 `代回群組：[內容]` 或 `Q-YYYYMMDD-HHMM-XX [內容]` |
| 可接受自然語法範例 | 「回覆這題：…」「幫我回群組：…」 |
| 小助手動作 | 通過三道檢查後，以 pushMessage 逐字轉貼至群組 |
| 是否公開回群組 | 是 |
| 是否需要確認 | **是**（高副作用；多 pending 時須指定短碼） |

代回前三道檢查：
1. 是哪一題（多 pending 須指定短碼；單筆可用「這題」）
2. thread 是否仍 open（已結案 / 群組 mute / OUT_OF_SERVICE_PERIOD / 服務期結束則拒絕；第一版不做 thread mute）
3. 權限（限 active admin 或 consultant）

## 附註

- 知識卡草稿不會自動生效；顧問「確認送出」→ Admin「確認更新」後才寫入 `knowledge_cards`。
- 待審草稿儲存於 `pending_knowledge_reviews` 表（review_id 格式 `K-YYYYMMDD-XX`）。
- REPLY_TO_GROUP 一律逐字轉貼，LLM 不得改寫顧問內容。
- 代回群組事件記錄於 event_logs，event_type 沿用 `consultant_override`，detail 含 `intent=REPLY_TO_GROUP`。
- 2-B 多步驟 DM session 已支援。2-C-1 私訊截圖理解已支援。2-C-2 screen_guides 畫面輔助線索、群組圖片公開回答等仍未實作。
