# 客立樂教學小助手實測問題整理｜2026-06-09

## 批次 1：私訊整理知識卡 UX 問題

### 1. 「結帳單操作」被誤判成帳務敏感

現象：
- 測試內容主要是在教店家如何新增結帳單。
- 系統因為文字中出現「結帳 / 帳務 / 金額 / 儲值」等字樣，將知識卡判定為敏感，導致驗證失敗。

初步判斷：
- 目前 validator 採關鍵字掃描，粒度太粗。
- 「新增結帳單 / 快速結帳單 / 臨時客結帳」屬於操作教學，不應直接等同帳務敏感。
- 使用者說「沒有涉及帳務」這類否定句，也不應反向觸發帳務風險。

建議修正方向：
- 將「結帳單操作」從帳務敏感中拆出，建立操作情境白名單。
- 敏感判斷改成更精準的語意/規則，例如金額錯誤、付款失敗、退款、對帳、發票、儲值扣抵異常才視為敏感。
- 增加否定語意處理，例如「沒有涉及帳務」不得觸發帳務風險。

### 2. 修改後仍重複驗證失敗

現象：
- 使用者輸入「修改：這張知識卡主要是教店家怎麼新增結帳單的操作，並沒有真的涉及帳務問題。」
- 系統仍重複回覆相同驗證失敗訊息。

初步判斷：
- validator 失敗時，草稿可能沒有被穩定保存為可繼續修改的狀態。
- 修改指令可能重新進入整理/驗證流程，但沒有明確針對上一份失敗草稿修正。

建議修正方向：
- validator 失敗時仍保留一份「待修正草稿」。
- `修改：...` 要能針對這份失敗草稿繼續修改，而不是重頭整理。
- 可以在 `dm_sessions.draft_data` 增加非狀態機欄位，例如 `validationStatus: failed`、`validationFailureReason`、`lastInvalidDraft`。

### 3. admin / consultant 指令提示混在一起

現象：
- 使用者是 admin，但草稿訊息提示「確認送出」。
- admin 輸入「確認送出」後，系統又提示 admin 應該輸入「確認更新」。
- admin 再輸入「確認更新」時，系統回覆目前沒有知識卡草稿。

初步判斷：
- 草稿提示文案沒有依身份分流。
- admin 與 consultant 共用同一組 action hints，導致使用者照錯指令。
- 驗證失敗後沒有有效可確認草稿，因此「確認更新」找不到草稿。

建議修正方向：
- admin 草稿提示只顯示：`確認更新 / 修改 / 取消 / 轉成 JSON`。
- consultant 草稿提示只顯示：`確認送出 / 修改 / 取消 / 轉成 JSON`。
- 驗證失敗但仍有待修正草稿時，admin 輸入「確認更新」應回覆「目前草稿尚未通過驗證，請先修改」，不要說沒有草稿。

### 4. 驗證失敗訊息重複洗版

現象：
- 系統連續多次回覆幾乎一樣的驗證失敗訊息。

初步判斷：
- 目前沒有明確的「驗證失敗待修正」互動狀態。
- 每次補充或修改都可能重新觸發完整失敗訊息。

建議修正方向：
- 第一次失敗回完整原因。
- 後續若仍在同一份草稿修正流程中，改用短回覆，例如「已收到修改，我會重新檢查這份草稿」。
- 避免同一原因連續重複推送。

### 5. 取消後仍跳出其他 fallback 訊息

現象：
- 使用者輸入「取消」後，預期只回覆「已取消目前知識卡整理流程，草稿資料已保留。」
- 實際又出現：
  - 「目前沒有草稿，請先『幫我整理知識卡』並提供內容。」
  - 「已收到。輸入『使用說明』查看可用指令，或『我的層級』查身份。」

初步判斷：
- 取消指令處理後，私訊 fallback 或無 session 指令處理仍被觸發。
- `修改：... / 補充：... / 確認更新 / 確認送出` 在沒有 active session 時，可能又進入一般 fallback。

建議修正方向：
- `取消` 指令只能回一則，不附加其他提醒。
- 取消後短時間內 suppress private fallback。
- 無 active session 時，`修改：... / 補充：... / 確認更新 / 確認送出` 應回覆明確單一訊息，不再觸發一般 fallback。

## 派工總方向

本批不是新增功能，而是修正現有私訊整理知識卡流程的 UX 與狀態銜接：
- 讓「操作教學」與「帳務敏感」不要被粗暴混在一起。
- 讓 admin / consultant 看到符合身份的下一步指令。
- 讓驗證失敗後可以自然修改同一份草稿。
- 避免重複洗版與取消後多重 fallback。

## 批次 2：知識卡命中後不公開回答、稍後處理待辦與使用說明

### 6. 已儲存「儲值卡設定」知識卡，但群組詢問時仍轉顧問

現象：
- admin 已確認更新並寫入知識卡 `001`。
- 匯出可看到該卡已存在。
- 店家在群組問「請問要怎麼儲值」時，小助手仍未公開回答，而是推問題收斂卡給顧問。

初步判斷：
- 目前這張卡被判定為不公開回答，訊息中顯示：
  - 小助手是否會自動公開回答：否
  - 涉及儲值 / 金額 / 帳務相關情境，僅作為導入教練參考
- 因此即使知識卡命中，`riskRouter` 仍會走 handoff，不會公開貼 standard_answer。
- 這是目前規則的結果，不一定是 knowledge match 失敗。

建議修正方向：
- 將「儲值卡設定 / 建立 / 新增 / 使用路徑」與「儲值金額異常 / 扣抵 / 餘額 / 付款 / 退款 / 對帳」拆成不同風險情境。
- 允許純操作教學型儲值卡知識卡成為可公開回答，但要保留敏感邊界：
  - 問設定路徑、建立方式、按鈕在哪裡：可公開回答。
  - 問金額錯誤、餘額異常、付款扣抵、帳務對帳：轉顧問。
- validator 不應只因「儲值卡」三字就強制不可公開，需看搭配語意。

### 7. 「稍後處理」後缺少待處理問題召回機制

現象：
- 顧問收到問題收斂卡後回覆「稍後處理」。
- 之後不容易叫回目前稍後處理 / 尚未回覆的問題卡。
- 現實上教練可能同時收到多個群組問題，需要知道哪些還沒處理。

目前可用但不夠直覺：
- 系統已有「查看待處理問題」指令，可列 open pending handoffs。
- 但問題收斂卡上的文案只有提示「稍後處理」，沒有明確告訴顧問之後如何叫回清單。

建議修正方向：
- 「稍後處理」回覆後應回一則確認訊息，例如：
  - 已先標記為稍後處理。之後可輸入「查看待處理問題」叫回未處理清單。
- 待處理清單需顯示：
  - 群組名稱
  - 問題短碼
  - 店家問題摘要
  - 目前狀態（open / 稍後處理）
  - 可用操作：代回、略過、建立/整理知識卡
- 代回確認時，確認卡不應只顯示問題編號，應同時顯示群組名稱、店家問題、即將代回內容。

### 8. 代回完成後是否引導建立知識卡

現象/需求：
- 若教練用代回方式解決群組問題，這段顧問回覆可能就是未來知識卡素材。
- 現在代回後沒有順手詢問是否整理成知識卡。

建議修正方向：
- 代回成功後，可私訊顧問：
  - 已代回群組。
  - 是否要把這題整理成知識卡草稿？可輸入「整理成知識卡」。
- 這不應自動建卡，只是進入既有私訊草稿流程。
- 仍需 admin / consultant 權限與既有確認流程。

### 9. 使用說明需隨功能更新同步維護

現象：
- 使用者在流程中輸入「使用說明」。
- 現有使用說明可能仍包含不直覺或已不建議使用的語法。

建議修正方向：
- 每次新增或調整功能，都要同步更新：
  - 私訊 Admin 使用說明
  - 私訊 Consultant 使用說明
  - 群組簡版使用說明
  - `CONSULTANT_MANUAL.md`
- 移除或降級不建議使用的語法。
- 加上待處理問題相關指令：
  - 查看待處理問題
  - 稍後處理
  - 指定短碼代回
  - 代回後整理成知識卡

## 批次 3：知識卡查詢 UX 問題

### 10. 查詢知識卡語法太死板，關鍵字查找沒有命中

現象：
- 使用者私訊輸入：
  - 查詢知識卡 儲值設定
  - 找跟儲值相關的知識卡
  - 列出所有知識哪
  - 列出所有知識卡
- 前幾個關鍵字查詢沒有回覆或沒有命中，只有「列出所有知識卡」成功。

初步判斷：
- 目前知識卡查詢指令可能只支援固定語法，例如「找跟登入有關的卡」或「列出所有知識卡」。
- 對顧問來說，查詢應該是自然語法，不應要求精準句型。
- 只要句子裡有「查 / 找 / 搜尋 / 知識卡」與關鍵字，就應該進入知識卡搜尋。

建議修正方向：
- 擴充查詢意圖辨識：
  - `查詢知識卡 [關鍵字]`
  - `找 [關鍵字] 相關知識卡`
  - `找跟 [關鍵字] 有關的卡`
  - `搜尋 [關鍵字]`
  - `有沒有 [關鍵字] 的知識卡`
- 對常見錯字/口語做容錯，例如「列出所有知識哪」可提示或視為「列出所有知識卡」。
- 查詢關鍵字應搜尋 title / patterns / standard_answer / not_applicable / escalate_to_consultant。
- consultant 仍只能查 active 卡；admin 可查全部。

### 11. 知識卡查詢結果不應顯示工程欄位

現象：
- 「列出所有知識卡」回覆：
  - `status=active risk=mid`
  - `created_by=...`
  - `confirmed_by=...`
  - `created_at=...`
- 這些內容對顧問不直覺，也無法知道目前可以怎麼回答店家。

初步判斷：
- 目前列表格式仍偏管理/工程資訊。
- 顧問查知識卡主要是想知道：
  - 這張卡在講什麼
  - 店家可能怎麼問
  - 小助手/顧問可以怎麼回答
  - 哪些情況不能用
  - 哪些情況要轉顧問

建議修正方向：
- 查詢結果改成人類可讀格式，尤其 consultant 版不顯示 userId / risk enum / raw status。
- 建議格式：
  - `【知識卡】儲值卡設定（001）`
  - `店家可能會問：...`
  - `建議回答：...`
  - `不適用：...`
  - `需要導入教練協助：...`
  - `小助手是否會自動回群組：會 / 不會，原因：...`
- admin 若需要追蹤欄位，可以用「查看知識卡詳細資料 [card_id]」或「匯出所有知識卡」查看，不應混在一般查詢結果裡。
